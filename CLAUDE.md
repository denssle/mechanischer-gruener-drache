## Bekannte Stolperfallen

- Callbacks/Handler, die als `void`-Funktion registriert werden (z.B. `webhookServer.onNotification(...)`/`onRevocation(...)` in `src/index.ts`) aber async sind, brauchen ein explizites `.catch()`. - Type-only Imports (z.B. `import { Command } from './types/discord.js'`) werden von `vitest`/esbuild nicht auf Existenz geprĂĽft, nur von `tsc`. Ein fehlendes Type-File fĂ¤llt also nicht bei `npm test` auf, sondern erst bei `npm run build` â€“ im CI-Workflow lĂ¤uft `npm run build` *nach* `npm test`, checkt also vor dem eigentlichen Fehlerfall niemand.

Bewusst nicht gefixt (Hobby-Projekt, seltener Edge-Case), falls es doch mal stĂ¶rt: Redis-SET mit kurzer TTL ĂĽber die Message-ID wĂ¤re der Ansatz.
- Jeder node-redis-Client **muss** einen `.on('error', ...)`-Listener haben, sonst crasht ein unhandled `'error'`-Event (z.B. bei einem Verbindungsabbruch/gescheiterten Reconnect) sofort den kompletten Bot-Prozess â€“ node-redis emittiert das aktiv, auch wĂ¤hrend der eingebauten automatischen Reconnect-Strategie. - `redisService.getSortedSet()` ist **hart auf die Top 10** begrenzt (`zRangeWithScores(key, 0, 9, {REV: true})`) â€“ gedacht fĂĽr Leaderboard-Anzeigen (Ping-Pong-Highscore), nicht zum Aufsummieren ĂĽber alle Mitglieder. - **Handler, die ĂĽber `commands/index.js` erreichbar sind** (also von einer `*.command.ts` importiert werden, z.B. `logging.handler.ts` via `log.command.ts`), dĂĽrfen `client` **nicht auf Modul-Top-Level verwenden** (z.B. `client.on(...)` direkt als Seiteneffekt beim Import).
