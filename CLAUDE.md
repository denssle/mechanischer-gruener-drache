# Mechanischer Grüner Drache

Discord-Bot für den Community-Server von [Legend of the Green Dragon (LotGD)](https://www.lotgd.de/), einem Browser-RPG. Hobby-Projekt für einen kleinen, privaten Server – Features wie Ping-Pong, Sport-Tracking und Twitch-Benachrichtigungen sind Community-Spielereien für diesen Server, nicht Teil des eigentlichen Spiels.

## Techstack

- TypeScript, Node.js 20, discord.js v14, Redis (Unix Socket)
- Hosted on Uberspace, Supervisor für Prozessverwaltung

## Struktur

- src/commands/ – Slash Command Definitionen
- src/handlers/ – Business Logik
- src/services/ – Redis und externe APIs
- src/server/ – Webhook Server (Twitch EventSub)
- src/types/ – Interfaces und Types

## Konventionen

- Deutsche Bot-Antworten
- **Emoji-Sparsamkeit (seit 2026-07-07, User-Feedback „visuelles Äquivalent von ADHS"):** Bot-Antworten sind bewusst (fast) emojifrei – keine führenden Status-Icons (✅/❌/⚠️), keine dekorativen/Fließtext-Emojis, auch die Zufalls-Flavor-Zeilen (Ping-Pong/Event/Sport) tragen keine mehr. Beim Bauen neuer Antworten also **keine Emojis reflexartig einstreuen**. Die früher erlaubten `/hilfe`- und Gruppen-`hilfe`-Kategorie-Icons wurden am 2026-07-10 auf User-Wunsch ebenfalls **entfernt** – die Hilfetexte (`hilfe.handler.ts` `HELP_TEXT` sowie `📖 **…-Befehle**` in `sport`/`twitch`/`event.handler.ts`) sind jetzt emojifrei. Verbliebene bewusste Ausnahmen: die Sport-Aktivitäts-Labels (`SportActivities`, dienen als Item-Icons), die Bestätigungs-Reaktion der Sport-Auto-Erfassung (`BESTAETIGUNGS_REAKTION` = ✅, ist dort die *einzige* Rückmeldung, kein Schmuck), und die **funktionalen** Statusmarker im Audit-Log (`logging.handler.ts`: 🗑️/➕/➖/🔇/🔊 zum schnellen Scannen im Log-Channel) sowie in `/twitch diagnose` (✅/⚠️ pro Subscription – reines Admin-Debugging). `console.log/warn`-Log-Ausgaben sind egal (nicht User-sichtbar).
- **Command-, Subcommand- und Options-Namen sind deutsch und sprechend** (z.B. `/twitch verknuepfen`, `/sport eintragen`, Option `benutzername`/`kanal`/`beschriftung`). Umlaute in den Namen selbst vermeiden – als `ae/oe/ue` schreiben (Discord mag keine Umlaute in Command-Namen); in den `setDescription`-Texten sind Umlaute normal. Am 2026-07-04 wurde der komplette Command-Bestand von gemischt englisch/deutsch auf durchgehend deutsch umgestellt (u.a. `ping`→`pingpong`, `pinghighscore`→`pingbestenliste`, `log`→`protokoll`, `twitch set/remove/info/notification-*`→`verknuepfen/entfernen/status/benachrichtigungs*`, `sport hinzufuegen/legacy`→`eintragen/altkilometer`). `version` blieb bewusst. Die Handler-Methodennamen wurden mitgezogen (z.B. `handleSet`→`handleVerknuepfen`), damit Dispatch und Namen konsistent bleiben.
- Command-Definitionen sind untypisierte Objektliteralle (`export default { data, execute }`), keine `satisfies Command` o.ä. Der `Command`-Typ (`src/types/discord.ts`) existiert nur für `client.commands`/`interaction.handler.ts`.
- Redis-Keys immer als KEYS-Objekt im Service. **Zwei dokumentierte Legacy-Ausnahmen** (Format bewusst nicht geändert, sonst verwaisen Bestandsdaten): der Ping-Pong-Score (`userId + REDIS_KEYS.PING_PONG`) und `user.service.ts` (speichert StoredUser unter der **rohen Discord-User-ID** ohne Prefix).
- **LotGD-Scraping immer über `lotgd.service.ts` → `fetchLotgdHtml(url, kontext)`** (seit 2026-07-10, Konsistenz-Durchgang): kapselt Bot-User-Agent (Standard-Fetch-UA bekommt `403`) + ISO-8859-1-Dekodierung + Fehler-Logging an einer Stelle. `news`/`online`/`character.service` nutzen ihn; neue LotGD-Features ebenso – den Fetch **nicht** erneut im Service ausschreiben. Netzwerkfehler reicht der Helper durch, die Services fangen sie im eigenen try/catch (Muster: `null` statt Crash).
- deferReply() bei API-Calls die länger dauern können
- Ephemere Antworten über `flags: MessageFlags.Ephemeral`, **nicht** das seit discord.js v14 deprecatete `ephemeral: true` (schrieb bei jeder Nutzung eine Deprecation-Warnung ins persistierte Log; am 2026-07-04 projektweit umgestellt).
- Tests mit Vitest, Redis wird pro Testdatei inline mit `vi.mock('../services/redis.service.js', () => ({ default: { ... } }))` gemockt (kein zentrales Mock-File, `src/mocks/` existiert nicht). Braucht die Mock-Factory eine Variable, `vi.hoisted()` benutzen (die Factory wird an den Dateianfang gehoben, normale Top-Level-`const`s existieren dort noch nicht).
- **Jeder Command hat einen `*.command.test.ts`** (seit Konsistenz-Durchgang 2026-07-10): flache Commands einen Delegations-Test (execute → Handler-Methode), Gruppen-Commands zusätzlich den **Drift-Test** (jeder im `SlashCommandBuilder` definierte Subcommand dispatcht auch im `switch` – die Namen stehen doppelt im Code, der Test ist die einzige Absicherung).

## Deployment

- `console.log`/`console.error`-Ausgaben werden vom Hoster (Uberspace/Supervisor) tatsächlich mitgeschnitten und persistiert – bei Fehlern die nur geloggt werden (z.B. gescheiterte Notification-Zustellung), ist das also keine reine Diagnose-Attrappe, sondern real nachvollziehbar. Kein zusätzliches Alerting nötig, solange man bereit ist, ins Log zu schauen.
- GitHub Actions (`.github/workflows/deploy.yml`) deployt bei jedem Push auf `main` **direkt in Produktion** auf Uberspace, kein Staging. `npm test` → `npm run build` → Twitch-Webhook-Integrationstest gegen den echten laufenden `dist/index.js` → `rsync` → `supervisorctl restart drache`. Ein rotes Gate verhindert das Deployment, ist also bewusst streng zu behandeln, nicht nur "CI grün bekommen".
- **Node-Version: CI und Uberspace-Host müssen zusammen wandern.** Die CI (`deploy.yml`, `setup-node`) läuft seit 2026-07-09 auf Node 22, vorher 20 (seit April 2026 End-of-Life). Weil der Host sich beim Restart selbst neu baut (siehe nächster Punkt), testet die CI nur dann wirklich das, was in Produktion läuft, wenn dort dieselbe Major-Version aktiv ist – auf Uberspace umstellbar per `uberspace tools version use node 22` (danach `supervisorctl restart drache`). Driftet das auseinander, ist das Gate grün, obwohl der Host-Build scheitern kann.
- **Zwei unabhängige „Node 20 ist deprecated"-Baustellen, nicht verwechseln:** (1) `node-version:` bei `setup-node` – die Version, mit der der *Bot* gebaut/getestet wird, muss zum Host passen (siehe oben). (2) Die Laufzeit der *Actions selbst* – `actions/checkout@v4`/`setup-node@v4` sind intern Node-20-Actions; GitHub zwingt sie auf Node 24 und schreibt eine Annotation ins Run-Log. Das behebt **nur** ein Bump auf `@v5` (seit 2026-07-09 gesetzt), nicht die `node-version`. Ein Bump von (1) macht die Warnung aus (2) also nicht weg.
- **Nicht auf Node 24 heben, solange Uberspace es nicht anbietet.** Die Obergrenze gibt der Host vor, nicht der Node-Release-Zyklus: `uberspace tools version use node 24` scheitert mit „Node.js version 24 does not exist", höchste verfügbare Version ist 22 (Stand 2026-07-09, prüfbar per `uberspace tools version list node`). 22 ist Maintenance LTS bis April 2027 – der nächste Bump wird also erst dann fällig bzw. sobald Uberspace 24 nachzieht.
- Supervisor auf dem Uberspace-Host startet den Bot über `npm run start-server` (= `npm ci && npm run build && npm start`) – der Server baut sich also bei jedem Restart selbst nochmal aus dem gerade gerynchten Source neu, das per CI vorgebaute `dist/` aus dem Workflow wird dabei überschrieben. D.h. `npm run build` muss nicht nur in der CI, sondern auch mit dem Node/npm auf Uberspace funktionieren.
- Skripte unter `scripts/*.ts` werden für CI/lokale Nutzung mit `tsconfig.scripts.json` zu `dist-scripts/` kompiliert und mit purem `node` ausgeführt (`npm run build:scripts`, `npm run test:twitch`). **Kein ts-node** – dessen ESM-Loader-Registrierung (`--esm`/`--transpile-only`) ist zwischen Node-Versionen/npx-Installationen unzuverlässig; hat lokal auf Node 24 funktioniert, in der Node-20-CI aber mit `ERR_UNKNOWN_FILE_EXTENSION` zuverlässig gecrasht.
- **Der Supervisor-Config von `drache` (`~/etc/services.d/drache.ini` auf dem Host) braucht `stopasgroup=true` + `killasgroup=true`.** Ohne das killt `supervisorctl restart` nur den `npm`-Wrapper (`npm ci && build && npm start`), der eigentliche `node dist/index.js`-Enkelprozess überlebt verwaist und **hält Port 3000 weiter belegt**. Der neu startende Prozess bekommt dann keinen sauberen Listener auf 3000 → der Webhook-Server ist von außen nicht erreichbar (`uberspace web backend list` zeigt `/twitch http:3000 => NOT OK, no service`, extern nginx `502`) → **Twitch-EventSub-Verifizierung scheitert, Subscriptions bleiben `webhook_callback_verification_pending`, es kommt gar keine Live-Meldung an**. Am 2026-07-04 live so gewesen (Testuser bekamen nichts), per sauberem Neustart (`supervisorctl stop` → `pkill -f dist/index.js` → `start`) behoben, `killasgroup` verhindert die Wiederkehr. Diagnose-Werkzeug dafür: `/twitch diagnose` (siehe Twitch-Feature) + `curl -i https://enzlor.uber.space/twitch/eventsub` (gesund = `404 Cannot GET`, kaputt = `502`).
- `webhookServer.start()` registriert seit 2026-07-04 einen `server.on('error', …)`-Handler (`handleServerError`), damit ein fehlgeschlagenes Port-Bind **laut ins (persistierte) Log** schreibt statt lautlos zu verschwinden – vorher lief der Bot bei EADDRINUSE scheinbar normal weiter (Discord ok), nur der Webhook war tot, was die Ursachensuche massiv erschwert hat. Analog zur node-redis-`error`-Falle (siehe Stolperfallen).

## Bekannte Stolperfallen

- Callbacks/Handler, die als `void`-Funktion registriert werden (z.B. `webhookServer.onNotification(...)`/`onRevocation(...)` in `src/index.ts`) aber async sind, brauchen ein explizites `.catch()`. Ohne das killt eine unhandled promise rejection den kompletten Bot-Prozess (Node ≥15) – ist im Twitch-Notification-Pfad am 2026-07-03 live passiert (Redis war nicht verbunden, `handleStreamOnline` warf, Prozess crashte).
- **`vitest.config.ts` grenzt die Suite auf `include: ['src/**/*.test.ts']` ein – das ist nicht kosmetisch.** Ohne die Datei sammelt vitest lokal **auch die nach `dist/` kompilierten Tests** ein (nach einem `npm run build` liegt dort von jeder `*.test.ts` eine `*.test.js`) und fährt die Suite **doppelt**: einmal gegen den Arbeitsstand, einmal gegen den Stand des letzten Builds. Man testet dann zur Hälfte alten Code, und die gemeldete Testzahl ist doppelt (am 2026-07-11 aufgefallen: „808 Tests", real sind es 409 in 41 Dateien). **In der CI fällt das nie auf**, weil dort `npm test` *vor* `npm run build` läuft und `dist/` noch leer ist – das Gate ist also grün, während lokal etwas anderes gemessen wird. Die Config nicht entfernen; kommt je ein zweites Ausgabeverzeichnis dazu (z.B. `dist-scripts/`), muss `include` weiterhin nur den Quellbaum umfassen.
- Type-only Imports (z.B. `import { Command } from './types/discord.js'`) werden von `vitest`/esbuild nicht auf Existenz geprüft, nur von `tsc`. Ein fehlendes Type-File fällt also nicht bei `npm test` auf, sondern erst bei `npm run build` – im CI-Workflow läuft `npm run build` *nach* `npm test`, checkt also vor dem eigentlichen Fehlerfall niemand.
- Twitch-EventSub-Notifications werden nicht per Message-ID dedupliziert – retried Twitch eine Zustellung (z.B. bei Timeout), könnte theoretisch zweimal "ist live"-gepostet werden. Bewusst nicht gefixt (Hobby-Projekt, seltener Edge-Case), falls es doch mal stört: Redis-SET mit kurzer TTL über die Message-ID wäre der Ansatz.
- Jeder node-redis-Client **muss** einen `.on('error', ...)`-Listener haben, sonst crasht ein unhandled `'error'`-Event (z.B. bei einem Verbindungsabbruch/gescheiterten Reconnect) sofort den kompletten Bot-Prozess – node-redis emittiert das aktiv, auch während der eingebauten automatischen Reconnect-Strategie. War in `redis.service.ts` bis 2026-07-03 nicht gesetzt; seitdem im Constructor registriert (nur Logging, kein Reconnect-Code nötig – node-redis regelt das selbst). Betraf potenziell den gesamten Bot, nicht nur Twitch.
- `redisService.getSortedSet()` ist **hart auf die Top 10** begrenzt (`zRangeWithScores(key, 0, 9, {REV: true})`) – gedacht für Leaderboard-Anzeigen (Ping-Pong-Highscore), nicht zum Aufsummieren über alle Mitglieder. Für "alle Werte summieren" (z.B. Sport-Gesamtkilometer) `getSortedSetAll()` benutzen (ungekürzt, `0, -1`). War in `sportService.getGesamtKilometer` bis 2026-07-03 falsch (undercounted ab >10 Score-Haltern, inkl. dem Legacy-Dummy-User).
- **Handler, die über `commands/index.js` erreichbar sind** (also von einer `*.command.ts` importiert werden, z.B. `logging.handler.ts` via `log.command.ts`), dürfen `client` **nicht auf Modul-Top-Level verwenden** (z.B. `client.on(...)` direkt als Seiteneffekt beim Import). Zirkuläre Kette `client.ts → commands/index.js → *.command.ts → handler.ts → client.ts`: `client` ist zu dem Zeitpunkt noch nicht initialisiert → `ReferenceError: Cannot access 'client' before initialization`. `twitch.handler.ts` nutzt `client` nur innerhalb von Methodenkörpern (funktioniert), `member.handler.ts` verkabelt sich selbst am Modul-Top-Level (funktioniert nur, weil es *nicht* über `commands/index.js` eingebunden wird, sondern direkt von `index.ts`). Handler, die von einem Command importiert werden UND `client.on(...)` brauchen, müssen die Verkabelung stattdessen in `index.ts` machen (wie `message.handler.ts`/`logging.handler.ts`). Am 2026-07-03 beim Bau des Logging-Features live so gecrasht, erst durch den Boot-Test in CI-Modus aufgefallen.

## Twitch-Feature (`/twitch`)

- 1 Discord-User = 1 Twitch-Link (`handleSet` blockt bei bestehendem Link). Channel + Rolle für Notifications sind global vom Admin gesetzt (`TWITCH:NOTIFICATION_CHANNEL`/`_ROLE`), nicht pro User.
- `twitchService.unsubscribeFromStreamOnline` behandelt ein 404 von Twitch (Subscription dort schon weg) als Erfolg – wichtig für Idempotenz bei `/twitch entfernen` und beim Revocation-Handling.
- Twitch `revocation`-Webhooks (z.B. Auth entzogen) räumen automatisch den zugehörigen Redis-Link auf (`twitchHandler.handleSubscriptionRevoked`, verkabelt über `webhookServer.onRevocation` in `src/index.ts`) – der User müsste sich danach neu verknüpfen, es bleibt aber kein verwaister Link liegen.
- `/twitch benachrichtigungskanal` ist über `addChannelTypes` auf Text-/Announcement-Channels beschränkt, damit kein Voice-Channel o.ä. gewählt werden kann, an dem `channel.send()` sonst still scheitern würde.
- `twitch.command.test.ts` prüft zusätzlich, dass jeder im `SlashCommandBuilder` definierte Subcommand-Name auch im `switch` in `execute()` dispatcht wird (die Namen stehen doppelt im Code, das ist die einzige Absicherung gegen Drift zwischen beiden).
- `/twitch diagnose` (Admin, seit 2026-07-04) ist das Debugging-Werkzeug für „es kommt keine Live-Meldung an": prüft ob ein Benachrichtigungskanal gesetzt **und abrufbar** ist, ob eine Rolle gesetzt ist, fragt via `twitchService.listStreamOnlineSubscriptions()` die EventSub-Subscriptions bei Twitch ab und zählt sie nach Status (nur `enabled` stellt zu – `webhook_callback_verification_pending` heißt: Twitch konnte den Webhook nie verifizieren, Endpoint nicht erreichbar oder `TWITCH_WEBHOOK_SECRET` falsch) und postet zum End-to-End-Test eine Nachricht in den Kanal. Antwort an den Admin ist ephemer, die Testnachricht landet real im Kanal.
- `handleStreamOnline` loggt seit 2026-07-04 an **allen** Abbruchstellen (unbekannter Broadcaster / kein Kanal / Kanal nicht abrufbar) mit `console.warn` – vorher waren zwei davon stille `return`s, was „keine Meldung"-Fehler unnötig schwer nachvollziehbar machte (Hoster persistiert die Logs).
- Die Live-Meldung nennt seit 2026-07-05 zusätzlich **Stream-Titel + Spiel/Kategorie**. Das `stream.online`-Event liefert die **nicht** mit (nur Broadcaster + `started_at`), daher ein separater Helix-`Get Streams`-Call (`twitchService.getStreamInfo(userId)` → `/streams?user_id=`). Bewusst **fehlertolerant**: `getStreamInfo` fängt alles ab und gibt `null` zurück, die Meldung baut Titel/Spiel als **optionale Zeilen** – kommt nichts zurück (Abruf-Fehler *oder* direkt beim Live-Gehen noch keine Daten bei Twitch), fällt sie einfach auf die alte Form zurück statt zu blockieren.

## Sport-Feature (`/sport`)

- Bewusst **kooperativ, nicht kompetitiv**: alle tragen zu einer gemeinsamen Gesamt-Kilometerzahl bei (`/sport gesamt`), es gibt keine Rangliste. Ein früherer `/sport bestenliste`-Befehl (Top-10-Leaderboard) wurde deshalb wieder entfernt (samt `sportService.getHighscore`) – nicht versehentlich, das war eine bewusste Design-Entscheidung gegen Konkurrenzdenken auf dem Server.
- `/sport eintragen` zeigt seit 2026-07-05 direkt in der Bestätigung die **neue gemeinsame Gesamtdistanz** (`getGesamtKilometer()` gleich mit abgefragt) – unterstreicht das kooperative Design (jeder Eintrag zahlt sichtbar auf die Gruppensumme ein), auf User-Wunsch.
- Die Eintrags-Bestätigung ist seit 2026-07-07 ein **Embed statt reinem Text** (auf User-Wunsch „persönlicher, damit man sich im Post wiederfindet"): `setAuthor` mit `interaction.user.displayName` + `displayAvatarURL()` (Name + Profilbild oben). Die **Eintrags-ID wanderte in den Footer** (bleibt für `/sport loeschen`/`bearbeiten` sichtbar). `handleEintragen` antwortet damit als einziger Sport-Handler mit `{embeds: [...]}` statt String – Tests prüfen entsprechend `reply.embeds[0].toJSON()`.
- **Auto-Erfassung im Sport-Kanal (seit 2026-07-13, Vorbild Blåhaj-Listener):** Schreibt jemand im Sport-Kanal eine normale Nachricht mit **`+`-markierter** Kilometer-Angabe („+12 km gelaufen"), trägt `sportHandler.handleMessage` (`MessageCreate`, eigene Registrierung in `index.ts` mit `.catch()`) den Eintrag automatisch ein und **quittiert nur per Reaktion** (`message.react(BESTAETIGUNGS_REAKTION)`, seit 2026-07-14 – siehe „Bestätigungen sind privat" unten). Vier bewusste Unterschiede zum Blåhaj-Listener: (1) **nur im konfigurierten Kanal** – serverweit würde jedes „noch 3 km bis zum Bahnhof" die gemeinsame Gesamtdistanz verfälschen; der Kanal ist bewusst **derselbe wie der Meilenstein-Ankündigungskanal** (`SPORT:ANNOUNCEMENT_CHANNEL`, gesetzt per `/sport ankuendigungskanal`) – kein zweiter Kanal-Befehl, ohne konfigurierten Kanal passiert gar nichts. (2) `parseKilometer` nimmt **nur die erste** km-Angabe (Blåhaj summiert alle) – doppelt gezählte Distanz würde die Gesamtsumme dauerhaft verfälschen – und verlangt seit **2026-07-14 zwingend ein `+`** direkt vor der Zahl (`/\+\s*(\d…)\s*(?:km|kilometer)\b/`, User-Wunsch): auch im Sport-Kanal wird über Kilometer *geredet* („die Strecke sind 12 km"), der Eintrag soll eine bewusste Geste sein. Ohne `+` passiert nichts. (3) `erkenneAktivitaet` rät die Sportart per Schlüsselwort, **ohne Treffer `DEFAULT_AKTIVITAET` = Laufen**. Die Muster sind **auf Wortgrenze verankert** (`\b`), nicht `includes`: „rad" steckt sonst in „Grad", „gerad" in „gerade" → beides würde als Radfahren eingetragen (Test nagelt das fest). (4) Bot-Nachrichten ignorieren ist auch hier **Pflicht** – die frühere Text-Bestätigung enthielt „12 km" und hätte sich selbst getriggert; der Check bleibt als Schutz stehen, falls je wieder geantwortet wird.
- **Auch bearbeitete Nachrichten werden erfasst (seit 2026-07-14):** `handleMessageUpdate` (eigene `MessageUpdate`-Registrierung in `index.ts` neben der des Loggings) läuft durch denselben Kern `erfasseKilometerAusNachricht` wie `handleMessage`. Grund: seit das `+` Pflicht ist, ist „Marker vergessen, Nachricht nachträglich korrigiert" der wahrscheinlichste Weg zum Eintrag. **Die Gefahr ist der Doppel-Eintrag** – Discord feuert `MessageUpdate` bei *jeder* Bearbeitung, `+5 km` → `+5 km, war schön` würde die 5 km sonst ein zweites Mal eintragen und die gemeinsame Gesamtdistanz dauerhaft verfälschen. Schutz ist die **eigene ✅-Reaktion** (`hatBereitsQuittung` → `reactions.cache.get(…)?.me === true`; `me` = *der Bot* hat reagiert, eine ✅ eines Mitglieds zählt nicht): hängt sie dran, wurde schon erfasst. Bewusst **kein zusätzlicher Redis-State** – die Quittung, die der Bot ohnehin setzt, *ist* die Markierung. Partielle Nachrichten (nicht gecacht) werden per `fetch()` nachgeladen, sonst wären `content` und `reactions` leer. Bewusst hingenommen: `+5 km` nachträglich auf `+50 km` zu ändern trägt **nicht** nach (dafür `/sport bearbeiten`), und wer die Reaktion von Hand entfernt und erneut bearbeitet, kann doppelt eintragen – das muss man absichtlich tun.
- **Bestätigungen sind privat (seit 2026-07-14, User-Wunsch „nur die eintragende Person soll sie sehen"):** `/sport eintragen` antwortet mit `flags: MessageFlags.Ephemeral` (das Embed bleibt, nur eben ephemer). Die **Auto-Erfassung kann das prinzipiell nicht** – ephemere Nachrichten hängen bei Discord an einem **Interaction-Token**, und eine normale Chat-Nachricht ist keine Interaction (dieselbe Mechanik, warum die Tipps nur an Slash-Commands hängen können). Deshalb dort statt einer Antwort nur eine **Reaktion** auf die Nachricht – der geräuschloseste Weg, der bleibt; die neue Gesamtdistanz nennt sie dafür nicht mehr (dafür gibt es `/sport gesamt`). Verworfen: Bestätigung per DM (kommt bei geschlossenen DMs gar nicht an). Der **Meilenstein-Post ist damit der einzige Sport-Post, den alle im Kanal sehen** – das ist gewollt, er ist der gemeinsame Moment.
- **Die Eintrags-ID ist aus der Bedienung verschwunden (seit 2026-07-13):** `/sport bearbeiten kilometer:<n>` und `/sport loeschen` (ganz ohne Option) arbeiten **immer auf dem zuletzt eingetragenen Eintrag** des Users (`sportService.editLastEntry`/`deleteLastEntry` → letztes Element der `SPORT:USER:<id>`-Liste, die per `rPush` gefüllt wird, also das neueste; `null` = User hat nichts eingetragen). Grund: eine UUID abzutippen war den Aufwand nie wert – korrigiert/gelöscht wird praktisch immer der eben gemachte Eintrag. `editEntry(userId, entryId, km)`/`deleteEntry` bleiben als Basis bestehen (die `*LastEntry`-Methoden benutzen sie). Folge: die **Eintrags-ID wird nirgends mehr angezeigt** – der Footer des Eintrags-Embeds ist weg, ebenso die ID in der Auto-Erfassungs-Antwort. Wird je wieder ein Zugriff auf ältere Einträge gebraucht, wäre eine Auswahl (Select-Menü/Buttons) der Weg, **nicht** die Rückkehr der UUID. `deleteLastEntry` gibt den gelöschten Eintrag zurück (nicht nur `boolean`), damit die Bestätigung nennen kann, was tatsächlich weg ist.
- **Kürze vor Flavor (seit 2026-07-13, User-Wunsch „so kurz und knapp wie möglich"):** Die Beschreibung im Eintrags-Embed ist **eine Zeile** (`Laufen – **10 km**, gemeinsam schon **250 km**.`), `/sport gesamt` ebenfalls (ohne `**Gesamtkilometer**`-Überschrift). Die früheren `EINTRAG_FLAVORS` (zufällige Motivationszeile, Pendant zu den Ping-Pong-Flavors) wurden dabei **ersatzlos entfernt** – beim Bauen neuer Sport-Antworten also keine Flavor-Zeilen/Überschriften wieder einführen. Ist dieselbe Linie wie die Emoji-Sparsamkeit.
- `/sport setzen` (Admin, Kilometerstand eines Users direkt setzen) schreibt nur den Bestenlisten-Score, legt aber keinen `SportEntry` an. Der eigene `/sport statistik`-Wert (aus den individuellen Einträgen berechnet) kann danach vom `/sport gesamt`-Beitrag dieses Users abweichen – als Admin-Korrekturwerkzeug für Altdaten so gewollt, aber gut zu wissen falls das mal für Verwirrung sorgt.
- `LEGACY_KILOMETERS` ist ein Dummy-User in der Bestenliste (`/sport altkilometer`, Admin) für Kilometer ohne zuordenbaren Discord-User – zählt zu `/sport gesamt` dazu, taucht aber nirgends als "Person" auf (seit Entfernung von `bestenliste` sowieso irrelevant, war vorher aber explizit rausgefiltert).
- `/sport altkilometer` **addiert** nur (`incrementSortedSet`) – es gab lange keinen Weg, die Bestandskilometer zu korrigieren oder zu entfernen. Seit 2026-07-04 gibt es dafür `/sport altkilometer-setzen kilometer:<n>` (Admin, `handleAltkilometerSetzen` → `sportService.setLegacyKilometer`), das den Legacy-Wert **direkt setzt** (statt zu addieren) und bei `0` den Legacy-Eintrag per neuem `redisService.removeFromSortedSet` (`zRem`) **ganz aus dem Sorted-Set entfernt**, statt einen 0-Score-Eintrag zu hinterlassen. Die Antwort nennt den vorherigen Wert (`getLegacyKilometer`). Betrifft bewusst **nur** die Bestandskilometer, nicht die individuell eingetragenen User-Kilometer – ein kompletter Gesamt-Reset ist bewusst nicht gebaut (siehe README-Todo).

### Meilensteine (seit 2026-07-07, war README-Todo „Sport-Event bei Schwelle")
- Zweck: Überschreitet die gemeinsame Gesamtdistanz eine Schwelle, postet der Bot einen frei formulierten Ankündigungstext (z.B. „Yay, wir haben 2000 km geschafft!") in einen konfigurierten Kanal. **Nichts mit dem `/event`-Feature zu tun** (das ist der Community-Termin-Countdown).
- Commands: **`/sport meilenstein setzen kilometer:<n> text:<text>` ist bewusst für ALLE offen** (User legen die Meilensteine an – so war der ursprüngliche Wunsch „von Usern angelegt"), `text` mit `\n`→echtem Umbruch + voller Markdown-Formatierung wie bei `/rollenbutton`. `/sport meilenstein liste` + `/sport meilenstein entfernen kilometer:<n>` + `/sport ankuendigungskanal kanal:<#channel>` sind **Admin**. `meilenstein` ist eine **Subcommand-Gruppe** (`getSubcommandGroup(false)` im Dispatch), `ankuendigungskanal` ein flacher Subcommand – der Drift-Test in `sport.command.test.ts` prüft entsprechend die Top-Level-Optionen (Gruppe zählt als eine).
- Speicherung: Redis-**Hash** `SPORT:MILESTONES`, Feld = Kilometerzahl (String), Wert = JSON `{kilometers, text, announced}` (neue Redis-Helper `setHashField`/`getHashAll`/`deleteHashField`). Kanal unter `SPORT:ANNOUNCEMENT_CHANNEL`.
- **`announced`-Flag pro Meilenstein** (User-Wunsch „einmal erreicht ist erreicht"): `checkAndMarkReachedMilestones()` nimmt `getGesamtKilometer()` (**alles zählt** – auch Legacy/Admin-gesetzte Werte, Admins müssen aufpassen), markiert alle noch nicht angekündigten, erreichten Meilensteine als `announced` und gibt sie zurück; das eigentliche Posten macht der Handler (braucht `client`).
- Verkabelt in **allen summen-ERHÖHENDEN** Handlern (`handleEintragen`/`handleBearbeiten`/`handleSetzen`/`handleAltkilometer`/`handleAltkilometerSetzen`), nach der jeweiligen Reply via privatem `announceReachedMilestones()`. **Nicht** in `handleLoeschen` (Verkleinern kann keine neue Schwelle auslösen; einmal gefeiert bleibt gefeiert).
- `announceReachedMilestones()` ist bewusst **fehlertolerant** (try/catch außenrum): darf eine erfolgreich gespeicherte Aktivität nie nachträglich scheitern lassen. **Ohne konfigurierten/abrufbaren Kanal wird `checkAndMarkReachedMilestones` gar nicht erst aufgerufen** → nichts wird als `announced` markiert → die Feier wird nachgeholt, sobald ein Kanal existiert und die Summe erneut steigt. (Rand-Fall: ist der Kanal da, aber `channel.send` scheitert, gilt der Meilenstein trotzdem als announced – wie bei Twitch bewusst nur geloggt.)
- `sport.handler.ts` importiert seit 2026-07-07 `client` (nur in Methodenkörpern, nie Top-Level → Zirkular-Import-Falle umgangen, wie `twitch.handler.ts`).

## Logging-Feature (`/protokoll`)

- `/protokoll kanal:<#channel>` (Admin) setzt den Ziel-Channel für Nachrichten-Logs, gespeichert in Redis (`LOGGING:CHANNEL`, `logging.service.ts`). Flacher Command ohne Subcommands (wie `/version`), da aktuell nur diese eine Aktion existiert. (Command am 2026-07-04 von `/log`/Option `channel` auf `/protokoll`/`kanal` eingedeutscht.)
- `client.ts` braucht `partials: [Partials.Message, Partials.Channel]`, sonst feuern `MessageDelete`/`MessageUpdate` für nicht (mehr) gecachte Nachrichten gar nicht erst – ohne das würde das Feature den Großteil der Deletes/Edits einfach verpassen.
- **Nachrichten-Cache (seit 2026-07-11):** Der alte Inhalt kam vorher nur aus dem RAM-Cache von discord.js (nach jedem Neustart leer, ältere Nachrichten immer `*nicht verfügbar*`). Jetzt schreibt `handleMessageCreate` (verkabelt in `index.ts` als **eigene** `MessageCreate`-Registrierung neben `messageHandler`/`blahajHandler`) Text + Autor-Tag + **Anhang-Dateinamen** nach `LOGGING:MESSAGE:<messageId>` mit **7 Tagen TTL** (`MESSAGE_CACHE_SECONDS`), Delete/Update lesen daraus, wenn discord.js nichts hat. Drei bewusste Grenzen: **nur wenn ein Log-Channel konfiguriert ist** (ohne `/protokoll` speichert der Bot weiterhin gar keinen Nachrichteninhalt), Eintrag wird **nach dem Lösch-Log sofort entfernt**, und Anhänge werden **nur namentlich** protokolliert (keine Dateien, keine CDN-Links – die sterben mit der Nachricht). Das ist eine bewusst getroffene Datenschutz-Entscheidung, kein Implementierungsdetail – dokumentiert in `docs/datenhaltung.md`, dort auch die vollständige Übersicht aller Redis-Daten. Neue Features, die Daten speichern, gehören in diese Tabelle.
- Die Änderungs-Erkennung bei `MessageUpdate` vergleicht seitdem die **aufgelösten** Inhalte (RAM-Cache *oder* Redis) statt `oldMessage.partial` als „unbekannt" zu behandeln – sonst würde ein Embed-Unfurling auf einer nur in Redis bekannten Nachricht fälschlich als Bearbeitung geloggt.
- Ist eine Nachricht älter als die TTL (oder war das Logging beim Verfassen aus), ist der alte Inhalt weiterhin nicht rekonstruierbar (Discord-API-Grenze, kein Bug) – zeigt dann einen Fallback-Text statt zu crashen.
- Bot-Nachrichten werden ignoriert (sonst loggt der Bot ständig seine eigenen Antworten). `MessageUpdate` ohne echte Content-Änderung wird ebenfalls ignoriert (Discord feuert das Event z.B. auch beim Nachladen von Link-Embeds, nicht nur bei echten Edits).
- **Beitritts-Zähler (seit 2026-07-11):** Die Beitritts-Meldung nennt bei Wiederkehrern, das wievielte Mal es ist („bereits zum 3. Mal") – Discord selbst merkt sich das nicht. Zähler in Redis (`MEMBER:JOIN_COUNT:{userId}`, dauerhaft, neuer `member.service.ts` + `redisService.increment` = `INCR`). Zwei bewusste Details: (1) Hochgezählt wird **in `handleGuildMemberAdd` selbst**, obwohl `member.handler.ts` auf dasselbe Event hört – `INCR` liefert den neuen Stand in einem Schritt zurück, während „in Handler A zählen, in Handler B lesen" ein Rennen zwischen zwei async-Listenern wäre. (2) Gezählt wird **vor** dem Log-Channel-Check, also auch ohne konfiguriertes Logging – sonst stünde die Zahl später falsch da. Beim ersten Beitritt wird sie **nicht** genannt (dort ist sie keine Information). Bestandsmitglieder starten bei 0: die Zählung beginnt ab jetzt, für alle Altbeitritte gibt es keine Daten.
- Loggt seit 2026-07-03 zusätzlich Server-Beitritt/-Austritt (`GuildMemberAdd`/`GuildMemberRemove`) in denselben Log-Channel. `member.handler.ts` hört auf **dieselben** Events für einen anderen Zweck (aktuell nur `console.log`, historisch war da mal mehr geplant) – zwei separate `client.on(...)`-Registrierungen für dasselbe Event in unterschiedlichen Dateien sind hier bewusst so, nicht redundant: unterschiedliche Zuständigkeiten (User-Daten-Tracking vs. Audit-Log-Channel).
- Loggt seit 2026-07-04 zusätzlich über **`GuildMemberUpdate`** (`handleGuildMemberUpdate`) drei Dinge in einem Durchlauf, die sich sonst überlappen: **Rollenänderungen** (Diff `newMember.roles.cache` vs. `oldMember.roles.cache` → `➕ … erhalten` / `➖ … verloren`), **Nickname-Änderungen** (`oldMember.nickname` vs. `newMember.nickname`, inkl. gesetzt/entfernt) und **Timeouts/Mutes** (`communicationDisabledUntilTimestamp` → `🔇 stummgeschaltet bis <t:…>` / `🔊 aufgehoben`). Alle Meldungen werden in eine `messages`-Liste gesammelt und nur bei nicht-leer der Log-Channel geholt/beschrieben. `GuildMemberUpdate` ist damit – wie GuildMemberAdd/Remove – **doppelt registriert**: `member.handler.ts` (`saveUser`, User-Daten) und `logging.handler.ts` via `index.ts` (Audit-Log), bewusst getrennt. Wichtig: Bei **nicht gecachtem** altem Mitglied (`oldMember.partial`) wird abgebrochen – sonst würde der leere Alt-Zustand jede bestehende Rolle fälschlich als "gerade erhalten" loggen. Kein zusätzliches Partial nötig, da `loadAllMembers()` beim Start alle Mitglieder cached (`GuildMembers`-Intent ist schon an). Rollenvergaben des Bots selbst (Button-Rollen) tauchen hier ganz normal als Änderung auf.
- Loggt seit 2026-07-04 außerdem **Bans/Unbans** (`GuildBanAdd`/`GuildBanRemove` → `handleGuildBanAdd`/`handleGuildBanRemove`, Grund wird mitgeloggt falls vorhanden) und **Massen-Löschungen** (`MessageBulkDelete` → `handleMessageBulkDelete`, postet Anzahl + Quell-Channel). Massen-Löschungen (z.B. Mod-Purge) feuern **nicht** als einzelne `MessageDelete`-Events, wurden vom Nachrichten-Logging vorher also komplett verpasst. Bans brauchen den zusätzlichen `GatewayIntentBits.GuildModeration`-Intent (`client.ts`); `MessageBulkDelete` läuft über den schon vorhandenen `GuildMessages`-Intent.

## Rollen-Selbstvergabe / Button-Rollen (`/rollenbutton`)

- Zweck: User geben sich **selbst** Rollen (z.B. "Einwohner" als Regel-Akzeptanz beim Server-Beitritt, oder die Twitch-Benachrichtigungs-Rolle zum Selbst-An/Abmelden). Am 2026-07-04 von einer Reaction-basierten Variante (`/rolle`, User reagiert mit Emoji) auf **Buttons** umgestellt – Buttons sind der modernere, sauberere discord.js-v14-Weg: kein Zumüllen der Nachricht mit Fremd-Reactions, klares ephemeres Feedback, selbsterklärendes Toggle. Die alte Reaction-Variante (`reactionRole.handler.ts`/`.service.ts`/`rolle.command.ts` + `REACTIONROLE:`-Redis-Keys) wurde dabei komplett entfernt, nicht parallel behalten.
- `/rollenbutton text:<text> rolle:<@Rolle> beschriftung:<beschriftung> [emoji:<emoji>]` (Admin) lässt den **Bot eine neue Nachricht posten** mit dem Text und einem Button darunter. Bot postet via `interaction.channel.send(...)`, quittiert dem Admin nur ephemer – die Button-Nachricht selbst ist also eine Bot-Nachricht (nötig, weil Discord Buttons nur an Bot-eigene Nachrichten erlaubt).
- Der `text` ist eine normale Bot-Nachricht → **volle Discord-Markdown-Formatierung** (fett, Überschriften, Zitate, …). Seit 2026-07-05 wandelt `handleCreate` literal getipptes `\n` per `.replaceAll('\\n', '\n')` in echte Zeilenumbrüche um – Slash-Eingabefelder erlauben kein echtes Enter, ohne das gäbe es keine mehrzeiligen Button-Nachrichten. Die **Button-`beschriftung`** ist dagegen **nicht** formatierbar (Discord-Grenze: Button-Labels sind reiner Text + optional ein `emoji`), kein Code kann das ändern.
- **Kein Redis, kein State:** Die Rolle steckt direkt in der Button-`customId` (`role-toggle:{roleId}`). Beim Klick liest `buttonRoleHandler.handleButton` die Rolle da raus und **toggelt** sie (User hat sie → entfernen, sonst → vergeben), ephemere Rückmeldung jeweils. Dadurch überlebt die Bindung jeden Bot-Neustart automatisch, ohne dass irgendwo etwas gespeichert werden muss. `reactionRole.service.ts` ist damit ersatzlos weggefallen.
- Verkabelung: Buttons sind **Interactions**, keine Reactions. `interaction.handler.ts` routet `interaction.isButton()` an `buttonRoleHandler.handleButton`. Dadurch fielen `GatewayIntentBits.GuildMessageReactions` + `Partials.Reaction`/`Partials.User` (`client.ts`) und die beiden `MessageReactionAdd`/`-Remove`-`client.on`-Zeilen in `index.ts` wieder weg – Buttons brauchen davon nichts.
- Setzt voraus, dass der Bot "Rollen verwalten"-Rechte hat **und** seine eigene Rolle in der Server-Hierarchie über der zu vergebenden Rolle steht – sonst kann `member.roles.add/remove` die Rolle nicht setzen. Bleibt reine Server-Konfiguration (kein Code kann fehlende Rechte erzeugen), **aber** seit 2026-07-04 prüfen sowohl `handleCreate` (beim Anlegen) als auch `handleButton` (beim Klick) das **proaktiv** (`me.permissions.has(ManageRoles)` + `me.roles.highest.comparePositionTo(role) <= 0`) und geben eine klare ephemere Meldung ("Rollen verwalten fehlt" / "Rolle steht über meiner in der Hierarchie") statt erst beim eigentlichen API-Call an einem generischen `Missing Permissions` zu scheitern. Der `handleButton`-catch bleibt als letztes Netz für alles andere.
- `buttonRole.handler.ts` importiert `client` **nicht** – die zirkuläre-Import-Falle (siehe Stolperfallen) fällt hier von vornherein weg. Die `client.on(...)`-Registrierung lebt ohnehin in `interaction.handler.ts` (das `client` importiert, aber *nicht* über `commands/index.js` erreichbar ist).

## Member-Handling (`src/handlers/member.handler.ts`)

- Einzige Handler-Datei, die (bis 2026-07-03) Logik direkt inline in `client.on(...)`-Callbacks statt als benannte Klassenmethoden hatte – deshalb ungetestet und beim Command-Block-Review leicht übersehen (kein Slash-Command). Jetzt wie die anderen Handler als `MemberHandler`-Klasse mit Methoden, `memberHandler` als Default-Export, `client.on(...)`-Verkabelung bleibt in der Datei selbst (nicht in `index.ts`, nur `loadAllMembers()` wird von dort aufgerufen).
- `handleUserUpdate` nutzt `client.guilds.cache.get(config.GUILD_ID)` statt `.first()` – der Bot ist aktuell ohnehin fest auf eine Guild ausgelegt (siehe README-Todo "Bot generalisieren für jeden Server"), aber `.first()` war eine implizite Annahme statt einer expliziten Referenz auf die konfigurierte Guild.

## Event-Feature (`/event`)

- Zweck (seit 2026-07-04, war README-Todo „Tage bis zum Treffen"): Ein Admin setzt einen Termin für das Community-Event, alle User können per Countdown „wie lange noch?" fragen. `/event setzen datum:<TT.MM.JJJJ> [uhrzeit:<HH:MM>] [titel:<text>]` (Admin), `/event countdown` (alle), `/event entfernen` (Admin).
- **Kein Datum-Objekt in Redis**, nur ein Unix-Timestamp (ms) + Titel als JSON unter `EVENT:NEXT` (`event.service.ts`). Genau **ein** Event gleichzeitig (Setzen überschreibt).
- Datum-Parsing (`parseGermanDateTime`, exportiert + separat getestet): deutsches Format `TT.MM.JJJJ` + optional `HH:MM`, gebaut mit `new Date(year, month-1, day, …)` (lokale TZ des Hosts = Europe/Berlin). **Wichtig:** danach wird per Round-Trip-Vergleich (`date.getDate() === day` etc.) validiert, weil `new Date()` ungültige Werte wie `32.13.` still auf den nächsten gültigen Tag normalisiert – ohne den Check würde Quatsch klaglos akzeptiert. Vergangene Daten werden beim Setzen abgelehnt.
- Anzeige nutzt **Discord-Timestamps** (`<t:unix:F>` absolut, `<t:unix:R>` relativ) – die lokalisieren sich pro Betrachter automatisch. Zusätzlich eine selbst ausgerechnete deutsche Restdauer (`formatRemaining`, exportiert + getestet): „X Tage und Y Stunden", Minuten nur wenn <1 Tag, mit korrektem Singular/Plural.
- Der `titel` ist **optional** und wird nur gespeichert, wenn angegeben (kein Default-String mehr). Der Countdown-Satz ist bewusst „**Noch {dauer}** bis zum nächsten Event{ – **titel**}!" formuliert: der Titel steht nach einem Gedankenstrich, **nicht** in einer Präposition – so entfallen Artikel-/Genus-Probleme (`bis zum {titel}` würde bei „die LAN-Party" o.ä. falsch), und die „Noch X"-Konstruktion umgeht das ist/sind-Agreement bei Singular vs. Plural der Restdauer.
- Ist **kein** Event gesetzt, antwortet `/event countdown` seit 2026-07-05 nicht mehr sachlich, sondern mit einer **zufälligen spielerischen Variante** von „Noch viel zu lange" (`randomNoEventReply()` aus `NO_EVENT_REPLIES`, beide exportiert + getestet) – bewusst im selben „Noch X"-Duktus wie der echte Countdown.
- `event.handler.ts` importiert `client` **nicht** (arbeitet nur mit `interaction`), Verkabelung läuft über `interaction.handler.ts` wie bei allen Slash-Commands – zirkuläre-Import-Falle also kein Thema.

## News-Feature (`/news` + `/ereignisse`)

- Zweck (seit 2026-07-04, war README-Todo): `/news` (alle) holt die neueste Spiel-News von `https://www.lotgd.de/news.php` **live per Scraping** und postet sie als einfachen Text (Titel, Datum, gekürzter Fließtext, Link zum Weiterlesen). Bewusst nur auf Abruf, kein automatisches Posten/Polling und kein State (die einfachste Variante war explizit gewünscht).
- **Es gibt keinen Feed/keine API** – nur HTML, und das ist **ISO-8859-1** kodiert. Der Abruf läuft über den geteilten `fetchLotgdHtml`-Helper (`lotgd.service.ts`, siehe Konventionen), der explizit mit `new TextDecoder('iso-8859-1')` dekodiert – sonst kommen kaputte Umlaute raus.
- Parsing ohne zusätzliche Dependency (`parseLatestNews`/`htmlToText`, exportiert + getestet): Jeder News-Eintrag ist als `<a name='motdJJJJMMTTHHMMSS'> … <hr></p>` eingefasst. Der `motd`-Anker liefert das Datum direkt (kein Locale-Parsing), das `</p>` ist die Ende-Grenze. Titel = erstes `<b>…</b>`, Body = alles hinter dem `colLtCyan`-Datums-Span bis zum `<hr>`. `<br>`→Zeilenumbruch, restliche Tags (v.a. LotGDs `<span class='colXXX'>`-Farbsoup) raus, gängige HTML-Entities dekodiert.
- **Fragil by design** (Scraping): ändert lotgd.de sein HTML/seine CSS-Klassen, liefert `parseLatestNews` `null` → Handler meldet „konnte nicht abrufen", crasht aber nicht. `getLatestNews` fängt zusätzlich Netzwerk-/Parse-Fehler ab und gibt `null` zurück. `handleNews` nutzt `deferReply()` (Netz-Call) und kürzt den Text auf 1500 Zeichen (Discord-2000-Limit).
- `news.handler.ts`/`news.service.ts` importieren `client` nicht – zirkuläre-Import-Falle kein Thema.
- **`/ereignisse` (seit 2026-07-09, war README-Todo „Neuigkeiten am"-Teil):** `news.php` hat **zwei** Abschnitte – oben die MOTD-Ankündigungen (`/news`), darunter unter der Überschrift `Neuigkeiten am <Datum>` das **Ingame-Ereignislog** (Spielgeschehen: wer wurde von wem getötet/wiederbelebt/blamiert). Beides holt `news.service.ts` aus **derselben** Seite (je Aufruf ein `fetchLotgdHtml`).
- Bewusst ein **eigener flacher Command** statt `/news`-Subcommand: sobald `/news` einen Subcommand bekäme, verlangt Discord *immer* einen, das blanke `/news` würde wegfallen (siehe Hilfe-Feature). Als flacher Command wird `/ereignisse` deshalb nur in `/hilfe` dokumentiert – `hilfe.handler.test.ts` sichert das per `it.each` ab.
- `parseGameEvents` (exportiert + getestet) schneidet **hinter dem `</div>` der Überschrift** an, nicht direkt hinter dem Datum: in derselben Überschrift steht noch der Paginierungs-Rest (`(Nachrichten 1 - 50 of 628)`), der sonst als erstes „Ereignis" im Chat landet. Ende-Grenze ist `<div id="petition">` (die Seitenleiste). Einträge sind durch `<div align='center'>…-=-=-…</div>`-Trenner separiert – einer davon trägt auf der echten Seite ein verirrtes `</span>` direkt hinter dem `<div>`, deshalb matcht der Trenner-Regex großzügig bis zum ersten `</div>` nach dem `-=-`.
- Das Überschriften-Datum ist **englisch** (`Thu, Jul 9, 2026`) und wird über eine Monats-Map ins deutsche `TT.MM.JJJJ` gebracht; unbekannte Formate bleiben unverändert stehen, statt zu scheitern.
- Die Seite liefert 50 Ereignisse, `/ereignisse` zeigt die **neuesten 5** (`EVENT_COUNT`). Beim 2000-Zeichen-Limit werden **ganze Ereignisse weggelassen** statt Sätze abgeschnitten (nur wenn schon das erste zu lang ist, wird es gekürzt) – Ereignisse sind vollständige Sätze, halbe wären unlesbar.

## Online-Feature (`/online`)

- Zweck (seit 2026-07-10, war README-Todo/Spiel-Anbindung): `/online` (alle) zeigt live per Scraping, wer gerade im Spiel eingeloggt ist. Quelle ist `https://www.lotgd.de/list.php` (die „Kriegerliste") – **öffentlich ohne Login abrufbar** (im Gegensatz zu Charakterdaten/Mail, die ein programmatisches Login bräuchten). Eigener Service `online.service.ts`, flacher Command wie `/news`/`/ereignisse` (nur in `/hilfe` dokumentiert, per `it.each` abgesichert).
- Wie News: **ISO-8859-1**, Bot-`User-Agent` (der Standard-Fetch-UA bekommt `403`), fehlertolerant. `getOnline()` macht **einen** Netz-Call und liefert daraus beide Listen.
- **Zwei Listen aus einer Seite** (Wunsch „beide kombiniert"): (1) `parseOnlinePlayers` = die reiche Haupttabelle „gerade eingeloggt" mit Spalten Gilde/Name/Ort/Level/Rasse/Geschlecht/Lebt (Geschlecht wird aktuell nicht angezeigt); (2) `parseRecentlyOnline` = die Seitenleiste „Online letzte 30 Minuten" (nur Namen). Der Handler hängt die 30-Min-Namen an, **dedupliziert** gegen die schon Eingeloggten.
- `parseOnlinePlayers` gibt **`[]`** zurück (niemand eingeloggt, Struktur ok) vs. **`null`** (Anker/Tabelle weg = Markup geändert) – der Handler unterscheidet: „niemand eingeloggt" vs. „konnte nicht abrufen". Tote Charaktere (`Lebt` = Nein) bleiben in der Liste, nur mit `(tot)` markiert. Namen sind auf der Seite oft Buchstabe-je-Farb-`<span>` zerlegt (Regenbogennamen) – `htmlToText` (aus `news.service.ts`, wiederverwendet) fügt sie zusammen.
- **Nebenfix am geteilten `decodeEntities` (news.service):** um die vollen Latin-1-Akzentbuchstaben (`&Uacute;`→Ú etc.) + hex-numerische Entities (`&#xDA;`) erweitert, weil Spielernamen sonst als `&Uacute;tlaga` durchrutschen. Kam `/news`/`/ereignisse` mit zugute.

## Charakter-Feature (`/charakter`)

- Zweck (seit 2026-07-10, war README-Todo „mit dem Spiel interagieren", lesender Teil): User verknüpfen ihren LotGD-Charakter und rufen dessen öffentliche Infos ab. **Bewusst „Weg A": nur der öffentliche Charaktername wird gespeichert, KEINE Zugangsdaten.** Grund: lotgd.de hasht das Passwort client-seitig per md5 und akzeptiert diesen Hash am `login.php` → der **Hash ist login-äquivalent**, „nur den Hash speichern" bringt keinen Sicherheitsgewinn. Für einen Hobby-Bot ist das gehortete Login-Geheimnisse Dritter = unverhältnismäßige Haftung. Details/Stufen 2+3 (private Daten, Aktionen – zurückgestellt): `docs/spiel-interaktion-idee.md`.
- Gruppen-Command (darf daher ein `hilfe`-Subcommand haben, konsistent mit `/sport`/`/twitch`): `verknuepfen name:` / `anzeigen [name:]` / `entfernen` / `hilfe`. `/hilfe` verweist im „Spielwelt"-Block auf `/charakter hilfe`. `/spielwelt` bleibt die Erklärung der drei **flachen** Info-Befehle (`/news`/`/ereignisse`/`/online`); `/charakter` ist eigenständig.
- Datenquelle: das **paginierte** öffentliche Roster `list.php?op=bypage&page=N` (~695 Charaktere, 100/Seite, auch offline). 8 Spalten inkl. **„Zuletzt da"**. `character.service.ts`: ISO-8859-1 + Bot-UA (wie news/online), `parseRoster` (`htmlToText` für Regenbogen-Namen, `null` bei kaputtem Markup), `getRoster` blättert bis zur ersten nicht-vollen Seite und **cacht in-memory** (`ROSTER_CACHE_MS` = 10 min; `clearCache()` erzwingt Neuabruf), sonst 7 Fetches pro Aufruf.
- **Titel-Präfix-Falle:** Der Roster zeigt „<Titel> <Name>" (z.B. „Centurio Acaine"), der Titel ändert sich mit Level/Drachentötungen. Gespeichert wird der eingegebene **Kern-Name**, `findInRoster` matcht **Suffix auf Wortgrenze** (exakter Name ODER endet auf „ Name") – sonst bräche die Zuordnung beim nächsten Titelwechsel. Randfall: zwei Charaktere mit gleichem Suffix → der erste gewinnt (bewusst simpel gehalten).
- Handler unterscheidet **„konnte nicht abrufen"** (`getRoster()` → `null`, Abruf/Markup kaputt) von **„nicht gefunden"** (`findInRoster` → `null`), deshalb ruft er beide getrennt auf statt einer Bequem-Methode. Karte ist ein **Embed** (Titel = aktueller voller Name, Farbe grün/dunkelrot nach lebendig/tot), analog zur `/sport eintragen`-Embed-Entscheidung. Beim `verknuepfen` wird der Name **gegen das Roster validiert** (Tippfehler abfangen); ein bestehender Link blockt (wie `/twitch`).
- `character.service.ts` hält zusätzlich `getAllLinks()` (Name ↔ Discord-User) als Basis für den Abgleich in `/online`/`/ereignisse` (siehe nächster Punkt). `character.handler.ts` importiert `client` nicht.

### Hervorhebung verknüpfter Charaktere in `/online` + `/ereignisse` (seit 2026-07-11)
- Taucht ein verknüpfter Charakter in der Online-Liste bzw. im Ereignislog auf, wird sein Name **fett** gesetzt und der Discord-User dahinter genannt (`<@id>`). Die Antworten gehen deshalb als **Objekt** raus (`editReply({content, allowedMentions: {parse: []}})`, vorher reiner String) – **`allowedMentions` ist Pflicht**, sonst pingt jedes `/online` alle verknüpften User an. Tests lesen entsprechend `editReply.mock.calls[0][0].content` (die reinen Fehler-/Leer-Antworten bleiben Strings).
- Zwei Matcher in `character.service.ts` (exportiert + getestet): **`findLinkForName`** für Anzeigenamen aus dem Spiel (Suffix-Match auf Wortgrenze wie `findInRoster`, wegen des Titel-Präfixes) und **`findLinksInText`** für den Ereignis-Fließtext. Letzterer benutzt Unicode-Lookarounds (`(?<![\p{L}\p{N}])…`) statt `\b` – `\b` kennt „Ú" nicht als Wortzeichen, „Útlaga" würde am Satzanfang nie matchen.
- **Fehlertolerant:** `getAllLinks()` läuft in beiden Handlern in einem eigenen try/catch (`links = []` im Fehlerfall) – ein Redis-Problem darf `/online`/`/ereignisse` nicht kosten, die Liste kommt dann nur ohne Hervorhebung.

## Hilfe-Feature (`/hilfe` + `hilfe`-Subcommands)

- Zweck (seit 2026-07-05, war README-Todo): Nutzer sollen ohne Vorwissen die Befehle finden. Zwei Ebenen: **Gruppen-Befehle** (`/sport`, `/twitch`, `/event`) haben je ein `hilfe`-Subcommand mit ihren eigenen Befehlen; **`/hilfe`** ist die neue **Gesamtübersicht** über alle Bereiche (flache Command-Datei `hilfe.command.ts` → `hilfe.handler.ts`, wie `/news`/`/version`).
- **Bewusste Design-Entscheidung:** flache Einzelbefehle (`/rollenbutton`, `/protokoll`, `/news`, `/version`, `/blahaj`, …) bekommen **kein** eigenes `hilfe` – sobald ein Command *einen* Subcommand hat, verlangt Discord *immer* einen (das blanke `/news` würde wegfallen). Sie werden deshalb **nur** in `/hilfe` erklärt. `hilfe.handler.test.ts` sichert per `it.each` ab, dass jeder dieser flachen Befehle im `HELP_TEXT` auftaucht (sonst wären sie nirgends dokumentiert).
- `HELP_TEXT` ist ein exportierter Konstanten-String (getestet: <2000 Zeichen fürs Discord-Limit, verweist für Gruppen auf `/sport|twitch|event hilfe`). Antworten sind **öffentlich** (konsistent mit den bestehenden `/sport hilfe`/`/twitch hilfe`), nicht ephemer.
- **`/spielwelt` (seit 2026-07-10)** ist die Detail-Hilfe für die drei Spiel-Daten-Befehle `/news`/`/ereignisse`/`/online`. Weil die flach sind und **kein** eigenes `hilfe`-Subcommand haben können (siehe Design-Entscheidung), bündelt dieser **eigene flache Befehl** ihre Erklärung – Parallele zu `/hilfe` (`spielwelt.command.ts` → `spielwelt.handler.ts`, exportierter `SPIELWELT_HELP`, öffentlich). `/hilfe` verweist im „Spielwelt"-Block auf `/spielwelt` (wie bei den Gruppen auf deren `hilfe`). `/spielwelt` ist selbst ein flacher Befehl → steht ebenfalls im `HELP_TEXT` und in dessen `it.each`-Absicherung.
- `hilfe.handler.ts`/`spielwelt.handler.ts` importieren `client` nicht – zirkuläre-Import-Falle kein Thema.

## Tipps & Nettigkeiten (`tipp.service.ts`, seit 2026-07-11)

- Zweck (war README-Todo „Tipps wie im Ladebildschirm"): Leute sollen Befehle entdecken, ohne `/hilfe` zu kennen. Dazu manchmal eine Zeile aus `TIPPS` – und, auf User-Wunsch, mit ~30 % Chance stattdessen eine aus `NETTIGKEITEN` (kleine Grüße ohne Belehrung, „Ich hoffe, du hast einen guten Tag").
- **Die Listen liegen als reine Daten in `src/data/tipps.ts` + `src/data/nettigkeiten.ts`** (17 bzw. 133 Zeilen), der Service re-exportiert sie. Das einzige Verzeichnis dieser Art im Projekt – gerechtfertigt allein durch die Länge: inline hätten die 100 Nettigkeiten die Auswahl-/Drossel-Logik im Service begraben. Bei kurzen Flavor-Listen (Ping-Pong, Sport, Event) bleibt es bewusst beim bisherigen Muster (inline im Handler, exportiert + getestet) – nicht vorschnell alles nach `src/data/` ziehen. Ein Test sichert die Listen gegen **Duplikate** ab (fällt beim Pflegen langer Listen sonst leicht durch, und eine doppelte Nettigkeit merkt man als Nutzer sofort).
- Ton der Nettigkeiten (bewusst): keine Aufgaben, keine Ratschläge im Befehlston, keine Leistungsbewertung – wer die Zeile liest, hat gerade einen Befehl benutzt und soll sich nicht ertappt fühlen. Emojifrei wie alles andere.
- **Die zentrale Design-Entscheidung ist, wo die Zeile landet:** als **ephemeres `followUp`** auf eine ohnehin ausgelöste Slash-Command-Antwort – **nicht** als eigener Post im Channel. Die Ladebildschirm-Analogie trägt nämlich nur halb: ein Ladebildschirm ist tote Zeit, ein Discord-Channel ist es nicht, dort kostet jeder Bot-Post die Aufmerksamkeit **aller** Anwesenden. Ephemer sieht die Zeile nur die Person, die gerade sowieso mit dem Bot interagiert – Clutter für den Rest ist damit strukturell ausgeschlossen (passt zur Emoji-/Noise-Linie des Projekts).
- Verkabelt **zentral** in `interaction.handler.ts` (`zeigeGelegentlichEinenTipp`, nach `command.execute()`), nicht in den einzelnen Handlern – sonst müsste man jeden Handler anfassen und bei jedem neuen Command dran denken. Bewusst **fehlertolerant** (try/catch): ein Tipp darf einen erfolgreich ausgeführten Befehl nie nachträglich scheitern lassen.
- Vier Drosseln, alle bewusst: `CHANCE = 0.15` (nicht bei jeder Antwort), Redis-**Tages-Cooldown** pro User (`TIPP:COOLDOWN:{userId}`, TTL 24 h – niemand liest denselben Hinweis dreimal am Abend), **keine Zeile an ephemeren Antworten** (das sind genau die Fehlermeldungen, Cooldown-Abfuhren und Admin-Quittungen – da löst jemand ein Problem und will kein Tutorial) und **keine bei `/hilfe`/`/spielwelt`** (dort ist sie redundant). Die Kontext-Regeln stecken in `kommtTippInFrage(commandName, warEphemer)` – bewusst rein/testbar, ohne Zufall und Redis.
- **Tipps nur zu nie benutzten Befehlen (seit 2026-07-12, war README-Todo):** `TIPPS` ist eine Liste von `{befehl, text}`-Objekten (Top-Level-Command-Name), `interaction.handler.ts` merkt sich **jede** Command-Ausführung per `tippService.merkeBenutztenBefehl` in einem Redis-Set (`TIPP:USED_COMMANDS:{userId}`, dauerhaft, nur Befehlsnamen – dokumentiert in `docs/datenhaltung.md`). Gemerkt wird **vor** allen Abbruchbedingungen (auch ephemere Antworten und `/hilfe` zählen als Benutzung). `randomTipp(benutzteBefehle)` filtert entsprechend; kennt die Person schon alle Tipp-Befehle, kommt stattdessen eine Nettigkeit (`?? randomNettigkeit()`), nie `null`-Stille nach gewonnenem Würfelwurf. Ein Test sichert ab, dass jeder `befehl` auch als `/name` im eigenen `text` vorkommt (sonst filtert der Abgleich an der falschen Stelle). Bestandsnutzer starten mit leerem Set – wie beim Beitritts-Zähler beginnt die Zählung ab Inbetriebnahme.

## Ping-Pong-Feature (`/pingpong`)

- **Ping-Pong ist seit 2026-07-11 reines PvP.** Das alte Solo-Spiel gegen den Bot (`/pingpong` als 50/50-Münzwurf, `handlePingPong` + `WIN_FLAVORS`/`LOSS_FLAVORS`) wurde **ersatzlos entfernt**, ebenso `/pingbestenliste` (samt `pingHighscore.command.ts`). Das war eine bewusste Entscheidung gegen die dokumentierte alte Design-Schwäche (rein additive Scores → die Bestenliste belohnte viel Klicken statt Spielen): Punkte gibt es jetzt **nur** noch aus Duellen gegen echte Personen, und ein Duell kostet den Verlierer auch etwas. Bestandsscores in Redis bleiben unangetastet gültig.
- Übrig sind drei Subcommands unter dem **Gruppen-Command** `/pingpong`: `herausfordern` / `bestenliste` / `hilfe` (Command-Datei heißt weiterhin `ping.command.ts`). Dass `/pingpong` eine Gruppe sein *muss*, ergibt sich aus der Discord-Regel: sobald ein Command *einen* Subcommand hat, verlangt Discord *immer* einen (siehe Hilfe-Feature) – ein blankes `/pingpong` konnte es neben `herausfordern` also gar nicht geben.
- `/pingpong bestenliste` zeigt die Top 10 nach Gesamtpunkten. Score-Key ist aus Legacy-Gründen `userId + REDIS_KEYS.PING_PONG` (Format bewusst nicht geändert, sonst verwaisen alle bestehenden Scores).
- **Anti-Spam-Cooldown** (seit 2026-07-05) – pro User ein Redis-Key mit TTL (`PING_PONG:COOLDOWN:{userId}`, `COOLDOWN_SECONDS = 30`, leicht anpassbar). Er hängt jetzt an `handleHerausfordern` (`redisService.getTimeToLive(...)`, ist noch Zeit übrig → ephemere „warte noch Xs"-Meldung, es wird **nicht** herausgefordert) und verhindert, dass jemand den halben Server in Serie herausfordert. Der **Herausgeforderte** bekommt keinen Cooldown – er hat das Duell nicht initiiert. Redis-Helper dafür: `setWithExpiry` (`SET ... EX`) und `getTimeToLive` (`TTL`, liefert -2/-1 = kein Cooldown aktiv).

### PvP-Duell (`/pingpong herausfordern gegner:@user`, seit 2026-07-11, war README-Todo)
- Ablauf: Der Bot postet eine **öffentliche** Nachricht mit zwei Buttons (Annehmen/Ablehnen). **Nur der Herausgeforderte darf klicken** (`interaction.user.id !== gegnerId` → ephemere Abfuhr). Bei Annahme wird das Match simuliert und die Nachricht per `interaction.update()` durch das Ergebnis ersetzt, **Buttons weg** – das ist zugleich der Schutz gegen mehrfaches Annehmen.
- **Kein Redis-State**: Herausforderer- und Gegner-ID stecken direkt in der `customId` (`pingpong-duell:{annehmen|ablehnen}:{herausfordererId}:{gegnerId}`), exakt wie bei den Button-Rollen → die Herausforderung überlebt einen Bot-Neustart, ohne dass etwas gespeichert werden muss. Nebeneffekt: Herausforderungen laufen nie ab (Discord-Buttons tun das nicht).
- Match: `spieleDuell()` (exportiert + getestet) spielt Ballwechsel à 50/50, bis einer `POINTS_TO_WIN = 3` hat – der Ausgang ist also reiner Zufall, nur mit mehr Dramatik als ein einzelner Münzwurf. Sieger **+1** (`DUELL_WIN`), Verlierer **−1** (`DUELL_LOSS`), aber `Math.max(0, …)` – **niemand kann ins Minus gespielt werden** (bewusst, Community-Bestenliste soll nicht bestrafen). Abschluss-Zeile aus `DUELL_FLAVORS` (exportiert + getestet).
- Selbst-Herausforderung und Bots werden ephemer abgelehnt (ein Bot hätte keinen sinnvollen Score).
- **Siegesserie (seit 2026-07-12, war README-Todo):** `verarbeiteSerie(siegerId, verliererId)` zählt die laufende Serie des Siegers per `redisService.increment` hoch (`PING_PONG:SERIE:{userId}`, INCR legt den Key selbst an und liefert den neuen Stand in einem Schritt) und **löscht** die des Verlierers. Der persönliche Rekord liegt bewusst in einem **eigenen** Key (`PING_PONG:REKORD:{userId}`), damit er die abgerissene Serie überdauert. Erzählt wird erst **ab `MIN_SERIE = 2`** – ein einzelner Sieg ist keine Serie, und ein „neuer Rekord" bei jedem ersten Sieg wäre Lärm (gespeichert wird die 1 trotzdem). Der Textbau liegt in der reinen Funktion `formatSerie(SerienStand)` (exportiert + getestet, `null` = nichts zu erzählen), das Duell-Ergebnis hängt sie nur an; `/pingpong bestenliste` zeigt die laufende Serie ab 2 hinter dem Punktestand.
- Verkabelung: Buttons sind Interactions – `interaction.handler.ts` ruft **beide** Button-Handler nacheinander auf (`buttonRoleHandler.handleButton` + `pingPongHandler.handleDuellButton`), jeder prüft seinen `customId`-Prefix selbst und steigt sonst sofort aus. `pingPong.handler.ts` importiert `client` **nicht** – zirkuläre-Import-Falle also kein Thema, obwohl der Handler über `commands/index.js` erreichbar ist.

### Ansage-Duell (`/pingpong ansageduell`, seit 2026-07-12, war README-Todo)
- **Bewusst ein eigener Subcommand neben `herausfordern`**, keine Option am bestehenden Duell (User-Wunsch): das normale Duell bleibt genau wie es war, die Variante steht daneben. Der **Befehl selbst ist die Ansage** („ich gewinne") – es gibt nichts zu wählen. Gespielt wird **unverändert per `spieleDuell()`** (reiner Zufall, die Ansage beeinflusst den Ausgang nicht).
- **Balancing (am 2026-07-12 direkt nach dem ersten Deploy nachgeschärft – 1.32.0 kannte nur den Bonus und beide Ansage-Richtungen):** Sieg/Niederlage wie immer (+1/−1), dazu **`ANSAGE_BONUS` (+1) bei erfüllter, `ANSAGE_MALUS` (−1) bei blamierter Ansage** – beides trifft nur den **Herausforderer**, denn nur er hat angesagt. Bonus und Malus sind **absichtlich gleich groß** (Erwartungswert null): eine reine Belohnung machte das Ansage-Duell strikt besser als das normale (im Schnitt +0,5 gratis) und hätte `herausfordern` verdrängt. Aus demselben Grund gibt es **keine Ansage „ich verliere"** – mit Bonus *und* Malus käme sie rechnerisch immer auf 0 heraus (−1+1 bzw. +1−1), also eine tote, risikofreie Option. Die **Siegesserie richtet sich weiter nach dem echten Spielausgang**, nicht nach der Ansage.
- Zustandslos wie das normale Duell: die Duell-Art steckt **im Prefix** der `customId` (`pingpong-ansage:{annehmen|ablehnen}:{herausfordererId}:{gegnerId}`, ansonsten identisch zum normalen Duell). `handleDuellButton` ist der Einstiegspunkt für **beide** Arten (`istAnsageDuell = prefix === ANSAGE_PREFIX`) – dadurch bleibt die Match-Auflösung an einer Stelle. `formatAnsage(istAnsageDuell, …)` (exportiert + getestet) baut die Ergebnis-Zeile, `null` beim normalen Duell.
- Der Malus geht als **negativer** `zusatzFuerVerlierer` in `trageErgebnisEin` – der `Math.max(0, …)`-Clamp greift dort deshalb **am Ende** der Rechnung, sonst könnte der doppelte Abzug jemanden ins Minus spielen.
- Der **Cooldown-Key ist derselbe** wie beim normalen Duell (`pruefeUndSetzeCooldown`, von allen Duell-Befehlen geteilt) – sonst könnte man abwechselnd über mehrere Befehle spammen.

### Taktikduell (`/pingpong taktikduell`, seit 2026-07-12, war README-Todo)
- **Ebenfalls ein eigener Subcommand** (User-Wunsch, wie beim Ansage-Duell): Schere-Stein-Papier im Ringschluss, `SCHLAEGT`-Map = Schmetterball→Lupfer→Konter→Schmetterball. `entscheideTaktik(herausforderer, gegner)` (exportiert + getestet) liefert `'herausforderer' | 'gegner' | 'gleich'`; bei **gleicher Aktion** fällt es auf den 50/50-`spieleDuell()` zurück (Satz wird dann mit angezeigt). Punkte und Siegesserie laufen unverändert.
- Zustandslos wie die anderen Duelle, aber mit einem **Kniff**: die Aktion des Herausforderers steht in der `customId` (`pingpong-taktik:{gegnerAktion|ablehnen}:{herausfordererId}:{gegnerId}:{herausfordererAktion}`), die des Herausgeforderten ergibt sich daraus, **welchen der drei Aktions-Buttons er klickt** – deshalb steht sie an der Stelle, an der die anderen Duelle `annehmen` haben. Vierter Button = Ablehnen.
- Eigener Einstiegspunkt `handleTaktikButton` (in `interaction.handler.ts` **zusätzlich** zu `handleDuellButton` verkabelt, jeder prüft sein Prefix selbst) – der Ergebnis-Aufbau unterscheidet sich zu stark, um ihn in `handleDuellButton` zu quetschen. Die geteilte Punkte-/Serien-Auswertung liegt in `trageErgebnisEin(siegerId, verliererId, bonusFuerSieger?, bonusFuerVerlierer?)`, die **alle drei Duell-Arten** benutzen.
- Bewusst hingenommen: die verdeckte Wahl ist im Client unsichtbar, aus dem API-Payload aber lesbar – für eine Community-Spielerei okay (stand schon so im README-Todo).

## Blåhaj-Rechner-Feature (`/blahaj` + Auto-Listener)

- Zweck (seit 2026-07-07, Community-Spielerei): Wird im Chat ein Euro-Betrag erwähnt, rechnet der Bot ihn in Blåhajs um (`EURO_PER_BLAHAJ = 28`) und summiert **alle je erwähnten Beträge** serverweit zu einer „Blåhaj-Fläche" in ha (`HECTARES_PER_BLAHAJ = 0.00003`). Beide Konstanten oben in `blahaj.handler.ts`, leicht anpassbar, falls sich der Preis ändert.
- **Zwei Einstiege:** (1) Auto-Listener auf `MessageCreate` – reagiert auf User-Wunsch **bei jeder** €-Erwähnung mit einer Antwort. (2) `/blahaj [betrag]` – ohne Betrag die Server-Gesamtsumme + Fläche, mit Betrag ein reiner Rechner. **Der `/blahaj betrag:`-Wert zählt bewusst NICHT zur Gesamtsumme** – nur der Auto-Listener akkumuliert (sonst wäre der Rechner nicht gefahrlos nutzbar).
- **Bot-Nachrichten werden zwingend ignoriert** (`message.author.bot`): sonst würde die „28 €" in der eigenen Antwort den Listener endlos erneut triggern. Das ist kein Nice-to-have, sondern verhindert eine Schleife.
- Verkabelung: **zweiter** `client.on(Events.MessageCreate, …)` in `index.ts` (neben `messageHandler.messageCreate`), bewusst getrennte Zuständigkeit wie bei den doppelt registrierten Logging-Events. Async void-Callback → `.catch()` (siehe Stolperfalle unhandled rejection). `blahaj.handler.ts` importiert `client` nicht (arbeitet nur mit `message`/`interaction`).
- Speicher: ein einzelner Redis-Float-Zähler `BLAHAJ:TOTAL_EUR` via neuem `redisService.incrementFloat` (`INCRBYFLOAT`, gibt String → `parseFloat`). Kein Verlauf pro User/Nachricht, nur die Summe.
- Parsing (`parseEuroAmounts`, exportiert + getestet): erkennt `50€`/`€50`/`50 €`/`50 Euro`/`50 EUR` inkl. deutschem Zahlenformat (Komma = Dezimal, Punkt = Tausender, mit Heuristik für alleinstehende Punkte). **Fragil by design** bei exotischen Formaten (z.B. „3€50" → nur 3) – bewusst simpel gehalten (Hobby-Spielerei), im Zweifel Tests erweitern.
- `/blahaj` ist ein **flacher** Command (kein Subcommand) → wie `/news`/`/version` nur in `/hilfe` dokumentiert (nicht per eigenem `hilfe`). `hilfe.handler.test.ts` sichert per `it.each` ab, dass `/blahaj` im `HELP_TEXT` steht.
